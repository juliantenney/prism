## 2) Purpose and Scope

### Purpose
Define the authoritative integration rules that connect Workstream 2 control outcomes to Sprint 46 decision-log recording, updates, and cross-linkage.

### Scope
- Applies to decision-log events generated by evidence packets, provenance controls, stage-exit controls, handoff validation, and escalation controls.
- Defines how entries are created, updated, superseded, cross-referenced, and linked to gates and escalation records.
- Ensures decision records remain traceable, auditable, and governance-aligned.

### Out of Scope
- Redefinition of governance authority structure, gate model, lifecycle stages, escalation semantics, or decision-log purpose.

---

## 3) Decision-Log Integration Framework

### 3.1 Integration Objective
Ensure all governance-significant Workstream 2 control outcomes are represented in `SPRINT-46-DECISION-LOG.md` with consistent structure and traceable linkage.

### 3.2 Integration Layers
1. **Event detection layer**  
   Identify control outcomes that require decision-log action.
2. **Entry action layer**  
   Decide create vs update vs supersede action.
3. **Linkage layer**  
   Attach references to source controls, artefacts, handoffs, escalations, and gate contexts.
4. **Verification layer**  
   Confirm entry completeness, consistency, and auditability.

### 3.3 Integration Principle
No gate-relevant or authority-impacting control event is treated as final until decision-log linkage is recorded where required by this specification.

---

## 4) Required Decision-Log Events

Decision-log integration is required for the following event classes:

1. Stage-exit outcomes with governance impact (`PASS WITH CONDITIONS`, `FAIL`, or authority-routed `PASS` path).
2. Handoff outcomes of `ACCEPT WITH CONDITIONS` or `REJECT`.
3. Escalation events requiring DA/governance adjudication.
4. Scope/authority interpretation outcomes affecting control disposition.
5. Provenance failures that affect acceptance or require authority adjudication.
6. Supersession of prior accepted handoff/exit/escalation decisions.
7. Governance-stage final disposition and next authorised action linkage.

## 6) Entry Creation Rules

A new decision-log entry must be created when:

1. No existing entry covers the same decision event context.
2. Event creates a new governance-significant decision point.
3. Event initiates authority-routed resolution path with no prior event record.

Each created entry must include:
- Event class
- Decision statement
- Rationale summary
- Outcome status
- Date/time and recorder role
- Source control artefact reference(s)
- Stage/handoff/escalation identifiers (as applicable)
- Gate reference (`Gate 0|Gate 1|Gate 2|Gate 3+|N/A`)
- Traceability references

---

## 7) Entry Update Rules

An existing decision-log entry must be updated when:

1. Event is a continuation of previously logged decision context.
2. Conditions, status, or adjudication outcome changes.
3. Remediation state changes from open to resolved/closed.

Update controls:
- Preserve original decision intent and historical context.
- Record updated fields with timestamp and updater role.
- Maintain explicit linkage to supporting evidence changes.
- Do not overwrite historical rationale; append update rationale.

---

## 8) Supersession Rules

Supersession applies when a prior logged decision is replaced by a materially revised disposition.

Supersession requirements:
1. Prior entry reference is mandatory.
2. Successor entry/reference is mandatory.
3. Supersession rationale must be explicit.
4. Effective date/time and author role must be recorded.
5. Supporting control artefact references for supersession must be linked.

Supersession integrity rule:
- Historical chain must remain auditable (`prior -> successor`) without deletion of prior entries.

---

## 9) Cross-Reference Requirements

Each decision-log-linked control record must cross-reference:
- control artefact ID/version,
- stage code (if stage-bound),
- handoff ID/version (if handoff-bound),
- escalation ID/severity/status (if escalation-bound),
- evidence packet ID/version,
- related provenance record reference(s).

Cross-reference quality rule:
- References must be resolvable and sufficient for independent re-check without hidden context.

---

## 10) Gate Linkage Requirements

Gate linkage is required whenever an event affects gate progression, gate interpretation, or gate-significant decisions.

Rules:
- Entry must include explicit gate reference (`Gate 0|Gate 1|Gate 2|Gate 3+|N/A`).
- Gate-relevant condition sets must include closure status and ownership.
- Governance-stage closure records must include final gate-context linkage and next authorised action reference.

---

## 11) Escalation Linkage Requirements

Escalation-linked decision entries must include:
- escalation reference ID,
- category and severity,
- routing path (stage-resolved or authority-routed),
- adjudication/closure outcome,
- handoff or stage-exit impact reference.

If escalation outcome changes handoff or stage disposition:
- decision-log update or supersession linkage is mandatory.

---

## 12) Traceability Requirements

Frozen traceability conventions apply:
- `S46-W1-<ARTEFACT>-<STAGE>-<SEQ>` for Workstream 1 semantic anchors.
- `S46-W2-<ARTEFACT>-<STAGE>-<SEQ>` for Workstream 2 controls.

Decision-log integration traceability requirements:
- Every decision-log-linked control event must carry traceable identifiers for artefact, stage, and decision context.
- Every entry action (create/update/supersede) must preserve predecessor/successor linkage where applicable.
- Control-to-decision-log linkage must remain reproducible from recorded references.

---

## 13) Auditability Requirements

Auditability controls:

1. **Completeness:**  
   All required decision-log fields and linkage references are populated.
2. **Consistency:**  
   Entry outcomes align with source control records (checklist/handoff/escalation/provenance/evidence packet).
3. **Chronology integrity:**  
   Create/update/supersede actions preserve temporal sequence and historical context.
4. **Reference resolvability:**  
   All cited artefact, handoff, escalation, and gate references resolve.
5. **Independent re-checkability:**  
   Another reviewer can reconstruct decision pathway from recorded links alone.

Audit failure conditions:
- missing linkage for required event classes,
- unresolved reference chains,
- overwritten history without supersession trail,
- contradiction between source control outcome and decision-log state.
